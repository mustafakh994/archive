## توثيق تشفير المرفقات (Attachment Encryption)

### نظرة عامة

- **الغرض**: تشفير ملفات المرفقات المخزّنة (at rest) لحمايتها في حال الوصول غير المصرّح به إلى التخزين.
- **الخوارزمية**: يتم استخدام **AES-256-GCM** (تشفير متناظر مع مصادقة).
- **التنسيق الموحّد بين الـ API وواجهة Next.js**:
  - البايتات المخزّنة تتبع البنية التالية:
    - `"AENC"` (4 بايت) = **Magic header**
    - **نسخة التنسيق** (1 بايت) = حاليًا القيمة `1`
    - **Nonce** عشوائي (12 بايت)
    - **Ciphertext** (حجم متغيّر)
    - **Tag** للمصادقة (16 بايت)
  - الصيغة النهائية:  
    \[
    \text{AENC (4)} + \text{Version (1)} + \text{Nonce (12)} + \text{Ciphertext (N)} + \text{Tag (16)}
    \]

### أين يتم التشفير وفك التشفير؟

- **في الـ API (ASP.NET Core)**:
  - الكلاس `AttachmentCryptoService` في `aspforms/Services/AttachmentCryptoService.cs`:
    - يستخدم `AesGcm` من .NET.
    - يقرأ المفتاح من الإعداد `AttachmentEncryption:KeyBase64`.
    - إذا كان المفتاح غير مضبوط (فارغ)، يتم تعطيل التشفير (`IsEnabled = false`) ويتم التعامل مع الملفات كنص عادي.
  - الإعدادات في الكلاس `AttachmentEncryptionSettings` في `aspforms/Configuration/AttachmentEncryptionSettings.cs`:
    - الخاصية `KeyBase64` يجب أن تكون **Base64 لمفتاح بطول 32 بايت** (AES-256).

- **في واجهة Next.js (forms_front)**:
  - الملف `forms_front/src/lib/attachment-crypto.ts`:
    - يطبّق نفس تنسيق الحزمة (`AENC` + version + nonce + ciphertext + tag).
    - يستخدم Node `crypto` مع الخوارزمية `'aes-256-gcm'`.
    - يشارك نفس المفتاح مع الـ API عبر متغيّر البيئة `ATTACHMENT_ENCRYPTION_KEY_BASE64`.

### من أين يأتي مفتاح التشفير؟

#### 1) في الـ API (aspforms)

- يتم ضبط المفتاح في ملفات الإعداد مثل:
  - `aspforms/appsettings.Development.json`
  - `aspforms/appsettings.Production.json`
  - `aspforms/appsettings.json`

- القسم الخاص بالتشفير:

```json
"AttachmentEncryption": {
  "KeyBase64": "<قيمة Base64 لمفتاح بطول 32 بايت>"
}
```

- مثال (بيئة التطوير الحالية):

```json
"AttachmentEncryption": {
  "KeyBase64": "hH3aCB4BnfctaWJMaOnCFIEg0DyUSNClbkAHCXM3VmE="
}
```

- في الإنتاج (`appsettings.Production.json`) القيمة حالياً فارغة:

```json
"AttachmentEncryption": {
  "KeyBase64": ""
}
```

> **ملاحظة**: عندما تكون القيمة فارغة، **يتم تعطيل التشفير** ويتم حفظ الملفات بدون تشفير.

#### 2) في واجهة Next.js (forms_front)

- يعتمد المفتاح على متغيّر البيئة:
  - `ATTACHMENT_ENCRYPTION_KEY_BASE64`
- في `attachment-crypto.ts`:
  - إذا كان المتغيّر مضبوطًا:
    - يتم فكّه من Base64 إلى `Buffer` بطول 32 بايت، وإذا لم يكن الطول 32 بايت يتم رمي خطأ.
  - إذا لم يكن مضبوطًا:
    - في **بيئة الإنتاج (`NODE_ENV === 'production'`)** → يتم رمي خطأ (`المفتاح إجباري في الإنتاج`).
    - في **بيئة التطوير** → يتم إنشاء ملف مفتاح تلقائي:
      - المسار: `.cache/attachment-dev.key`
      - يحتوي على مفتاح عشوائي (32 بايت) مشفّر بـ Base64.
      - يتم استخدام نفس الملف تلقائيًا في كل مرة ما دام موجودًا.

### كيف تعمل عملية التشفير وفك التشفير؟

#### في الـ API (`AttachmentCryptoService`)

- **التشفير**:
  1. التحقق أن `_key` غير فارغ (وإلا يتم رمي `InvalidOperationException`).
  2. إنشاء `nonce` عشوائي بطول 12 بايت.
  3. إنشاء كائن `AesGcm` بالمفتاح وطول الـ tag (16 بايت).
  4. استدعاء `Encrypt(nonce, plaintext, ciphertext, tag)`.
  5. بناء الحزمة النهائية:
     - كتابة `"AENC"` في أول 4 بايت.
     - كتابة `FormatVersion` (حاليًا `1`) في البايت الخامس.
     - نسخ الـ `nonce` (12 بايت) بعد ذلك.
     - نسخ الـ `ciphertext` بعد الهيدر.
     - نسخ الـ `tag` في النهاية.

- **فك التشفير**:
  1. التحقق أن `_key` غير فارغ.
  2. التحقق من صحة الحزمة عن طريق:
     - `LooksLikeEncryptedPackage`:
       - الطول ≥ `HeaderSize + TagSize`.
       - أول 4 بايت تساوي `"AENC"`.
       - البايت الخامس يساوي `FormatVersion`.
  3. استخراج:
     - `nonce` من الموقع (5 .. 5+12).
     - طول الـ ciphertext من: `package.Length - HeaderSize - TagSize`.
     - `ciphertext` من بعد الهيدر وحتى ما قبل الـ tag.
     - `tag` من آخر 16 بايت.
  4. استخدام `AesGcm.Decrypt(nonce, ciphertext, tag, plaintext)`:
     - إذا تم العبث بالبيانات أو كانت غير صحيحة → يتم رمي `CryptographicException`.

#### في Next.js (`attachment-crypto.ts`)

- **التشفير (`encryptAttachmentBuffer`)**:
  1. استدعاء `resolveKeyBuffer()` للحصول على مفتاح بطول 32 بايت.
  2. إنشاء `nonce` عشوائي بطول 12 بايت.
  3. إنشاء `cipher = crypto.createCipheriv('aes-256-gcm', key, nonce)`.
  4. بناء `encrypted = cipher.update(plain) + cipher.final()`.
  5. الحصول على `tag = cipher.getAuthTag()`.
  6. إرجاع `Buffer.concat([MAGIC, [FORMAT_VERSION], nonce, encrypted, tag])`.

- **فك التشفير (`decryptAttachmentBuffer`)**:
  1. الحصول على المفتاح بنفس الطريقة.
  2. التحقق من:
     - الحد الأدنى للطول (`HEADER_SIZE + TAG_SIZE`).
     - أول 4 بايت = `"AENC"`.
     - البايت الخامس = `FORMAT_VERSION`.
  3. استخراج:
     - `nonce` من البايتات (5 .. 5+NONCE_SIZE).
     - `tag` من آخر 16 بايت.
     - `ciphertext` ما بين الهيدر وآخر 16 بايت.
  4. إنشاء `decipher = crypto.createDecipheriv('aes-256-gcm', key, nonce)` مع `decipher.setAuthTag(tag)`.
  5. إرجاع النص المفكوك: `decipher.update(ciphertext) + decipher.final()`.

### كيف أغيّر مفتاح التشفير؟

> **تحذير مهم**: تغيير المفتاح يعني أن **جميع الملفات التي تم تشفيرها بالمفتاح القديم لن يمكن فك تشفيرها بالمفتاح الجديد**.  
> إذا كنت بحاجة للحفاظ على الملفات القديمة:
> - إمّا أن تقوم بعملية **إعادة تشفير (re-encryption)**:  
>   1) فك الملفات القديمة بالمفتاح القديم.  
>   2) إعادة تشفيرها بالمفتاح الجديد.  
> - أو تخزين أكثر من مفتاح (غير مدعوم حالياً في هذا الكود، ويحتاج تطوير إضافي).

#### 1) توليد مفتاح جديد (32 بايت) بصيغة Base64

- مثال باستخدام `openssl` (من أي جهاز آمن):

```bash
openssl rand -base64 32
```

- الناتج سيكون نص Base64 (مثلاً):

```text
Qklc0Qb8FJ4g1R9rYlLfC1f1gJpR3u2RkzZ5C8nO2kU=
```

- تأكد أن:
  - الطول بعد فك Base64 هو **32 بايت** بالضبط.

#### 2) تغيير المفتاح في الـ API (aspforms)

- افتح ملف الإعداد المناسب:
  - للتطوير: `aspforms/appsettings.Development.json`
  - للإنتاج: `aspforms/appsettings.Production.json`

- عدّل القيمة داخل قسم `AttachmentEncryption`:

```json
"AttachmentEncryption": {
  "KeyBase64": "Qklc0Qb8FJ4g1R9rYlLfC1f1gJpR3u2RkzZ5C8nO2kU="
}
```

- ثم:
  - **أعد تشغيل** تطبيق الـ API حتى يقرأ القيمة الجديدة.

#### 3) تغيير المفتاح في واجهة Next.js (forms_front)

- يجب أن تستخدم نفس المفتاح تمامًا المستخدم في الـ API، لضمان التوافق في التشفير/فك التشفير.

- في بيئة الإنتاج:
  - اضبط متغيّر البيئة `ATTACHMENT_ENCRYPTION_KEY_BASE64` بنفس القيمة:

```bash
export ATTACHMENT_ENCRYPTION_KEY_BASE64="Qklc0Qb8FJ4g1R9rYlLfC1f1gJpR3u2RkzZ5C8nO2kU="
```

  - أو أضِفه في ملف البيئة الخاص بـ الإنتاج حسب إعدادات النشر لديك (Docker, hosting, إلخ).
  - **أعد تشغيل** خدمة Next.js (أو الـ Node process) بعد التعديل.

- في بيئة التطوير:
  - إذا كنت تريد استخدام نفس المفتاح الموجود في الـ API (موصى به عندما تحتاج لاختبار end-to-end):
    1. اضبط متغيّر البيئة `ATTACHMENT_ENCRYPTION_KEY_BASE64` في `forms_front` (مثلاً في `.env.development`):

```env
ATTACHMENT_ENCRYPTION_KEY_BASE64=Qklc0Qb8FJ4g1R9rYlLfC1f1gJpR3u2RkzZ5C8nO2kU=
```

    2. احذف - إذا رغبت - ملف المفتاح التلقائي القديم:

```bash
rm -f forms_front/.cache/attachment-dev.key
```

    3. أعد تشغيل مشروع `forms_front`.

  - إذا **لم** تضبط متغيّر البيئة:
    - سيستمر النظام في إنشاء/استخدام الملف `.cache/attachment-dev.key` بمفتاح عشوائي للتطوير فقط.
    - هذا المفتاح لن يطابق مفتاح الـ API إلا إذا قمت بنسخه يدويًا إلى إعدادات الـ API.

### تمكين وتعطيل التشفير

- **في الـ API**:
  - التشفير مفعّل فقط عندما:
    - `AttachmentEncryption:KeyBase64` قيمة صحيحة (Base64 → 32 بايت).
  - إذا كانت القيمة فارغة أو غير صحيحة:
    - `IsEnabled = false`
    - أي استدعاء لـ `Encrypt` أو `Decrypt` سيرمي استثناء.

- **في Next.js**:
  - الدالة `isAttachmentEncryptionEnabled()` ترجع:
    - `true` إذا كان `ATTACHMENT_ENCRYPTION_KEY_BASE64` مضبوطًا.
    - في التطوير: `true` دائمًا (لأنه يتم إنشاء/استخدام مفتاح dev تلقائي).
    - في الإنتاج بدون متغيّر بيئة → يتم رمي خطأ عند محاولة الاستخدام (المفتاح مطلوب).

### خطوات عملية لتغيير المفتاح بشكل آمن

1. **خطّة للتعامل مع الملفات القديمة**:
   - قرّر إن كنت ستفقد إمكانية فك تشفيرها، أو ستنفّذ عملية إعادة تشفير.
2. **توليد مفتاح جديد آمن** (32 بايت Base64).
3. **تحديث إعدادات الـ API** في ملف `appsettings.*.json` المناسب.
4. **تحديث متغيّر البيئة في forms_front** (`ATTACHMENT_ENCRYPTION_KEY_BASE64`) بنفس القيمة.
5. **إعادة تشغيل** الـ API وواجهة Next.js.
6. **اختبار شامل**:
   - رفع ملف جديد والتأكد أنه يُحفَظ ويُستعاد بنجاح.
   - محاولة تنزيل المرفقات القديمة (إن كانت لا تزال مستخدمة) للتأكد من أنك لم تفقد بيانات مهمة.

### الفرق بين التشفير المتناظر وغير المتناظر ولماذا اخترنا المتناظر

#### التشفير المتناظر (Symmetric Encryption)

- **عدد المفاتيح**: مفتاح واحد فقط.
- **الاستخدام**: نفس المفتاح يُستخدم لـ **التشفير** و**فك التشفير**.
- **المزايا**:
  - **سريع جدًا** وكفؤ في التعامل مع الملفات الكبيرة (مرفقات، صور، PDFs…).
  - حجم البيانات الناتجة قريب من حجم البيانات الأصلية (فقط إضافة `nonce + tag` صغيرة).
  - أبسط في الإدارة عندما تتحكم في جميع الأطراف (Backend + Frontend لنفس النظام).
- **التحدي**:
  - يجب حماية المفتاح وعدم تسريبه؛ لأن من يملك المفتاح يستطيع التشفير وفك التشفير معًا.

#### التشفير غير المتناظر (Asymmetric Encryption)

- **عدد المفاتيح**: زوج مفاتيح:
  - **Public key**: يمكن مشاركته مع الآخرين.
  - **Private key**: يبقى سريًا.
- **الاستخدام النموذجي**:
  - تشفير باستخدام الـ Public key وفك التشفير باستخدام الـ Private key.
  - أو توقيع رقمي باستخدام الـ Private key والتحقق باستخدام الـ Public key.
- **المزايا**:
  - مناسب جدًا لـ **تبادل المفاتيح بأمان** بين أطراف لا تثق ببعضها (مثل ما يحدث في TLS/HTTPS).
  - يدعم سيناريوهات التوقيع الرقمي وعدم الإنكار.
- **القيود في سياق الملفات الكبيرة**:
  - أبطأ بكثير من التشفير المتناظر.
  - غير عملي لتشفير ملفات كبيرة مباشرة؛ غالبًا يُستخدم فقط لتشفير مفتاح متناظر، ثم يتم استخدام هذا المفتاح المتناظر لتشفير البيانات.

#### لماذا اخترنا التشفير المتناظر (AES‑256‑GCM) في هذا المشروع؟

- **١) السيناريو هو “تشفير ملفات عند التخزين” (Encryption At Rest)**:
  - المرفقات تُرفع إلى السيرفر وتُخزَّن مشفّرة في التخزين الخلفي.
  - نفس النظام (Backend + Frontend التابع له) هو من يقوم بالتشفير وفك التشفير.
  - لا يوجد طرف خارجي مجهول يحتاج فقط مفتاحًا عامًا، لذلك **لا نستفيد كثيرًا من Public/Private هنا**.

- **٢) الأداء مهم مع المرفقات**:
  - AES‑256‑GCM متناظر وسريع جدًا، ومناسب لتشفير وفك تشفير ملفات بحجم كبير.
  - التشفير غير المتناظر سيكون عنق زجاجة لو استُخدم مباشرة على الملفات.

- **٣) البساطة في الإعداد والإدارة**:
  - لدينا مفتاح واحد:
    - في الـ API: `AttachmentEncryption:KeyBase64`.
    - في الـ frontend: `ATTACHMENT_ENCRYPTION_KEY_BASE64` (أو ملف dev).
  - هذا يقلل من تعقيد إدارة المفاتيح، ويوضح التوثيق لزملاء الفريق.

- **٤) لا يوجد حالياً احتياج لخصائص التشفير غير المتناظر**:
  - لا نحتاج لتوقيع رقمي لكل ملف من مستخدم بمفتاحه الخاص.
  - لا نحتاج لتبادل مفاتيح بين كيانات لا تثق ببعضها.
  - لا نحتاج لسيناريو “طرف يقوم بالتشفير دون القدرة على فك التشفير”.

- **الخلاصة**:
  - التشفير المتناظر بـ **AES‑256‑GCM** هو الأنسب هنا لأنه:
    - يوفر **سرعة عالية**.
    - يناسب **تشفير الملفات عند التخزين** في نظام واحد متحكم به.
    - يحافظ على **بساطة التصميم** (مفتاح سري واحد مشترك بين مكوّنات النظام).
  - يمكن مستقبلاً إضافة طبقات أخرى (مثل توقيع رقمي أو إدارة مفاتيح أكثر تعقيدًا) عند الحاجة، دون تغيير جوهر هذا التصميم.
